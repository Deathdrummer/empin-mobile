# DDRSpecs Framework

Микрофреймворк для управления задачами с экономией токенов и сохранением контекста между сессиями.

## Принципы

1. **Одна активная задача** — не распыляемся, контролируем контекст
2. **Минимум токенов** — читаем только нужные файлы (DISPATCHER → конкретная task)
3. **Git интеграция** — каждая задача = отдельная ветка
4. **Handoff** — можно продолжить в новом треде с того же места
5. **Документирование** — встроенная система документации компонентов/сервисов
6. **Логирование** — автоматическое отслеживание и исправление ошибок

## Структура

```
.claude/ddrspecs/
├── AGENTS.md                       # Этот файл (инструкции)
├── ACTIVE.md                       # Указатель на активную задачу
├── HISTORY.md                      # Глобальный контекст выполненных задач
├── .logspath                       # Пути к файлам логов
├── .do-session                     # Сессия одноразовой задачи (временный)
├── run-session-log.json            # Лог выполненных задач (временный)
├── docs/                           # Документация проекта
│   ├── INDEX.json                  # Индекс документации
│   ├── components/                 # Компоненты
│   ├── services/                   # Сервисы
│   ├── models/                     # Модели
│   ├── controllers/                # Контроллеры
│   ├── utils/                      # Утилиты
│   └── architecture/               # Архитектурные решения
└── {task-slug}/                    # Папка задачи
    ├── REQUEST.md                  # История запроса и решений
    ├── SPEC.md                     # Описание задачи
    ├── DISPATCHER.md               # Карта задач + статусы
    └── tasks/
        ├── 001-{name}.md           # Подзадача
        ├── 002-{name}.md
        └── ...
```

**Временные файлы (в .gitignore):**
- `.do-session` - slug активной одноразовой задачи (только для `/ddrspec:do`)
- `run-session-log.json` - лог выполненных задач в текущей сессии (только для `/ddrspec:run`)

## ACTIVE.md

Содержит slug активной задачи:
```
drag-and-drop-filemanager
```

Если пустой или отсутствует — нет активной задачи.

## DISPATCHER.md — формат

```markdown
# {Task Name}

## Meta
- Slug: {task-slug}
- Branch: feature/{task-slug}
- Created: {date}
- Status: planning|in_progress|done
- Request: [REQUEST.md](./{slug}/REQUEST.md)

## Checkpoint
- Current: {номер текущей задачи или 0}
- Last Updated: {timestamp}

## Tasks
| # | File | Status | Title |
|---|------|--------|-------|
| 1 | 001-research.md | done | Исследование |
| 2 | 002-implement.md | current | Реализация |
| 3 | 003-test.md | pending | Тестирование |

## Context (кратко)
- Выбран: {решение/библиотека из REQUEST.md}
- Подход: {краткое описание из REQUEST.md}
- Ключевые файлы: {список}
- Потенциальные риски: {список из REQUEST.md, если есть}
```

**Статусы задач:**
- `done` — выполнено
- `current` — в работе (только одна!)
- `pending` — ожидает
- `blocked` — заблокировано

## REQUEST.md — формат

**Создаётся автоматически при `/ddrspec:new`**

```markdown
# Запрос на выполнение задачи

## Оригинальный запрос пользователя
{исходный текст}

## Консультация и корректировки
{анализ, альтернативы, выбор пользователя}

## Уточняющие вопросы и ответы
{полная история диалога}

## Финальная формулировка задачи
{скорректированная формулировка}

## Технологический стек
{выбранные технологии с обоснованием}

## Потенциальные проблемы и решения
{предвиденные проблемы}
```

**Назначение:** Сохранить весь контекст принятия решений для изолированного выполнения задач.

**ВАЖНО:** REQUEST.md - файл-справочник. НЕ читай его при каждой задаче! Вся критичная информация должна быть в task файлах.

## Task файл — формат

```markdown
# Task {N}: {Title}

## Цель
{Что нужно сделать — 1-2 предложения}

## Контекст
{ТОЛЬКО критичная информация для ЭТОЙ задачи - макс. 5-7 строк}
{Краткая выжимка из best practices - 1-2 предложения}

## План
1. [ ] Шаг 1
2. [ ] Шаг 2
3. [ ] Шаг 3

## ⚠️ Потенциальные проблемы
{ТОЛЬКО если критично - макс. 2-3 строки}

## Прогресс
{Заполняется по мере выполнения}

## Результат
{Заполняется после выполнения}

## Проблемы и решения
{Заполняется через /ddrspec:bug или /ddrspec:logs при возникновении проблем}

---
📄 Полный контекст: [REQUEST.md](../REQUEST.md) (читай только при проблемах)
```

**ЭКОНОМИЯ ТОКЕНОВ:**
- Task файл должен быть КОМПАКТНЫМ (50-100 строк максимум)
- НЕ дублируй информацию из REQUEST.md
- Пиши только то, без чего НЕЛЬЗЯ выполнить задачу

### 🐛 Проблема {N}: {название}
**Дата**: {YYYY-MM-DD HH:MM}
**Описание**: {описание проблемы}
**Решение**: {описание решения}
**Изменено**: {список файлов}
**Источники**: {ссылки на GitHub/StackOverflow}
```

## Команды

### Основные команды

**Выбор между `/ddrspec:new` и `/ddrspec:do`:**

| Параметр | `/ddrspec:new` | `/ddrspec:do` |
|----------|----------------|---------------|
| **Структура** | Полная (DISPATCHER, tasks, REQUEST) | Минимальная (только ветка) |
| **ACTIVE.md** | Записывается slug | Не изменяется |
| **Ветка** | `feature/{slug}` | `feature/do-{slug}` |
| **Продолжение** | Можно в новых тредах (`/ddrspec:run`) | Только в одном треде |
| **Декомпозиция** | 3-7 подзадач | Без декомпозиции |
| **Коммиты** | Можно частично (`/ddrspec:commit`) | Только в `/ddrspec:done` |
| **Консультация** | Проактивный анализ запроса | Проактивный анализ запроса |
| **Завершение** | `/ddrspec:done` (универсальный) | `/ddrspec:done` (универсальный) |
| **Когда использовать** | Сложные задачи 4+ файла | Простые задачи 1-3 файла |

### /ddrspec:new {описание}

**Консультативное создание задачи с проактивным анализом:**

1. **Глубокий анализ запроса** (КРИТИЧЕСКИ ВАЖНО):
   - Анализ на антипаттерны и устаревшие подходы
   - Поиск более простых/лучших решений
   - Выявление потенциальных проблем
   - Предварительный поиск через Context7/WebSearch

2. **Консультация пользователя** (если нужно):
   - Предложение альтернативных best practice решений
   - Советы по выбору технологий
   - Предупреждения о рисках

3. **Уточняющие вопросы** (AskUserQuestion):
   - С рекомендациями "(Рекомендую)" и "(Best Practice)"
   - Проверка выбора пользователя после каждого ответа

4. **Поиск конкретных технологий** (Context7, WebSearch)

5. **Сохранение REQUEST.md**:
   - Оригинальный запрос
   - Вся история консультации
   - Все уточняющие вопросы и ответы
   - Выбранные технологии с обоснованием
   - Потенциальные проблемы и решения

6. **Создание структуры**:
   - Папка `.claude/ddrspecs/{slug}/`
   - REQUEST.md (обязательно!)
   - SPEC.md с ссылкой на REQUEST.md
   - DISPATCHER.md с мета-данными из REQUEST.md
   - tasks/ с ссылками на REQUEST.md
   - Git ветка `feature/{slug}`

**Интеграция с документацией:**
- При упоминании `@ComponentName` читается `.claude/ddrspecs/docs/INDEX.json`
- Если компонент задокументирован → загружается документация
- Если нет → предлагается создать через `/ddrspec:doc-set`

**Золотые правила:**
- Всегда анализировать запрос на проблемы
- Всегда предлагать альтернативы
- REQUEST.md обязателен для каждой задачи

### /ddrspec:do [{описание}]

**Быстрая одноразовая задача в одном треде без создания полной структуры.**

**Режимы работы:**

1. **С аргументом** - `/ddrspec:do "описание"` - создание новой задачи
2. **Без аргумента** - `/ddrspec:do` - продолжение после смены аккаунта

**Когда использовать:**
- ✅ Простые задачи (1-3 файла)
- ✅ Быстрые правки/фиксы
- ✅ Задачи в одном треде
- ✅ Не требует детальной декомпозиции

**Когда НЕ использовать:**
- ❌ Сложные задачи (4+ файла)
- ❌ Требуется несколько тредов
- ❌ Нужна детальная декомпозиция
- ❌ Задача может занять несколько сессий

**Алгоритм (с аргументом):**
1. Проверить наличие `.claude/ddrspecs/.do-session` (предупредить если есть)
2. Проверить git статус и ACTIVE.md (предупредить о конфликтах)
3. Анализ запроса (@mentions, уточняющие вопросы, best practices)
4. Создать временную ветку `feature/do-{slug}`
5. Зарегистрировать сессию: `echo "{slug}" > .claude/ddrspecs/.do-session`
6. Выполнить задачу
7. Самопроверка (без багов, антипаттернов, инлайн-стилей)
8. Вывести список изменений и остановиться
9. **НЕ делать git commit** (будет в `/ddrspec:done`)

**Алгоритм (без аргумента - продолжение):**
1. Проверить наличие `.claude/ddrspecs/.do-session`
   - Если НЕ существует → ошибка "Нет активной задачи"
2. Прочитать slug из файла
3. Переключиться на ветку `feature/do-{slug}`
4. Проанализировать контекст треда:
   - История сообщений
   - Git изменения (diff, status)
   - Определить что сделано и что осталось
5. Продолжить выполнение задачи с того места где остановились

**Механизм сессий (.do-session):**
- Файл создаётся при запуске `/ddrspec:do "описание"`
- Содержит slug задачи
- Остаётся до вызова `/ddrspec:done`
- Предотвращает завершение задачи в другом треде
- Позволяет продолжить работу при смене аккаунта

**Важно:**
- Не изменяет ACTIVE.md
- Не создает DISPATCHER/tasks/REQUEST
- Завершается только через `/ddrspec:done` В ТОМ ЖЕ ТРЕДЕ
- Для продолжения в новых тредах используй `/ddrspec:new`

### /ddrspec:run

**КРИТИЧНО:** Экономия токенов!

1. Прочитать ACTIVE.md → получить slug
2. Если пусто → "Нет активной задачи. Создай через /ddrspec:new"
3. Прочитать DISPATCHER.md → найти `current` задачу
4. Прочитать ТОЛЬКО файл текущей задачи
5. **НЕ ЧИТАЙ REQUEST.md** - вся нужная информация в task файле
6. Выполнить задачу
7. Обновить статус в DISPATCHER.md:
   - Текущую → `done`
   - Следующую → `current`
   - Обновить Checkpoint
8. Если все задачи done → вывести "Все задачи выполнены. Запусти /ddrspec:done"

**REQUEST.md читай ТОЛЬКО если:**
- В task файле явно не хватает информации
- Возникла проблема, требующая понимания полного контекста
- Нужно вспомнить, почему выбрали конкретное решение

**Интеграция с логами:**
- В конце выполнения предлагается проверить логи через `/ddrspec:logs`

### /ddrspec:list

1. Прочитать все папки в `.claude/ddrspecs/`
2. Для каждой прочитать DISPATCHER.md (только Meta и Tasks)
3. Вывести таблицу:
```
# DDRSpecs Tasks

| Slug | Status | Progress | Branch |
|------|--------|----------|--------|
| drag-and-drop | in_progress | 2/4 ✓ | feature/drag-and-drop |
| auth-refactor | done | 5/5 ✓ | feature/auth-refactor |
```
4. Пометить активную задачу: `→ drag-and-drop (active)`

### /ddrspec:bug {описание проблемы}

Решить проблему в текущей задаче:

1. Прочитать ACTIVE.md → slug текущей задачи
2. Прочитать DISPATCHER.md → найти current task
3. Прочитать ТОЛЬКО файл текущей задачи
4. Анализировать проблему (логи из `.logspath`, документация из `docs/INDEX.json`, @mentions)
5. Поиск решения (WebSearch, Context7)
6. Решить проблему
7. Обновить task файл — добавить секцию "Проблемы и решения":
   ```markdown
   ## Проблемы и решения

   ### 🐛 Проблема {N}: {название}
   **Описание**: {problem}
   **Решение**: {описание решения}
   **Изменено**: {список файлов}
   **Источники**: {ссылки}
   ```
8. Если решение критично → обновить Context в DISPATCHER.md
9. Git коммит: `fix({slug}): {описание}`
10. Обновить Checkpoint в DISPATCHER.md

**Когда использовать:**
- Ошибка при выполнении задачи
- Баг в коде
- Неожиданное поведение библиотеки
- Нужно изменить подход внутри задачи

### /ddrspec:commit

**Коммит выполненных задач без merge в main.**

**Только для обычных задач (с ACTIVE.md)!**

1. Проверить ACTIVE.md (если пустой → СТОП)
2. Прочитать `.claude/ddrspecs/run-session-log.json`
3. Если лог пустой → "Нет выполненных задач"
4. Проверить git изменения
5. Сформировать коммит:
   - Одна задача: `feat({slug}): выполнена задача {num} - {title}`
   - Несколько: `feat({slug}): выполнены задачи {nums} - {titles}`
6. Выполнить коммит и очистить лог
7. Вывести результат с предложениями продолжить

**Когда использовать:**
- Нужно закоммитить прогресс без завершения всей задачи
- Несколько задач выполнено, но не все
- Безопасная точка для передачи контекста

### /ddrspec:done

**Универсальное завершение задачи (обычной или одноразовой).**

**Определение типа задачи:**
1. Проверить ACTIVE.md
   - Если содержит slug → **обычная задача** (Блок A)
2. Если ACTIVE.md пустой → проверить текущую ветку
   - Если ветка `feature/do-*` → **одноразовая задача** (Блок B)
3. Иначе → ошибка "Нет активной задачи"

**Блок A: Обычная задача (с DISPATCHER)**
1. Загрузить DISPATCHER.md → проверить все задачи выполнены
2. Если не все done → спросить подтверждение
3. Проверить session log → предложить коммит
4. Проверить git статус → закоммитить незакоммиченные изменения
5. Merge в main: `git merge {branch} --no-ff`
6. Обновить DISPATCHER.md (Status: done, Completed: дата)
7. Очистить ACTIVE.md и session log
8. Удалить feature ветку
9. Финальный коммит в main
10. Вывести результат

**Блок B: Одноразовая задача (do)**
1. Проверить наличие `.claude/ddrspecs/.do-session`
   - Если НЕ существует → ошибка "Нельзя завершить из другого треда"
2. Определить slug из ветки `feature/do-{slug}`
3. Проанализировать изменения → описание
4. Коммит: `feat: {description}`
5. Merge в main: `git merge {branch} --no-ff`
6. Очистка сессии: удалить `.do-session` и закоммитить
7. Удалить feature ветку
8. Обновить HISTORY.md (опционально, если файл существует)
9. Вывести результат

**Важно:**
- Git команды выполнять строго последовательно
- Обрабатывать конфликты merge
- Предлагать `/ddrspec:done` повторно после разрешения конфликтов

**Сравнение завершения:**

| Параметр | Обычная задача | Одноразовая задача |
|----------|----------------|-------------------|
| Триггер | ACTIVE.md содержит slug | Ветка `feature/do-*` + `.do-session` |
| DISPATCHER.md | Проверяется и обновляется | Не существует |
| ACTIVE.md | Очищается | Не изменяется |
| Session log | Проверяется и очищается | Не используется |
| Session файл | Не используется | `.do-session` (проверяется!) |
| HISTORY.md | Не обновляется | Обновляется (опционально) |
| Коммит | `feat({slug}): ...` | `feat: {description}` |
| Продолжение | Любой тред | Только тот же тред |

### /ddrspec:redo

Отметить текущую задачу как нерешенную и вернуться к предыдущей задаче.

1. Прочитать ACTIVE.md → получить slug
2. Прочитать DISPATCHER.md → найти current task
3. Спросить причину через AskUserQuestion
4. Обновить DISPATCHER.md:
   - Текущую задачу → `pending`
   - Предыдущую задачу → `current`
   - Обновить Checkpoint (Checkpoint.Current -= 1)
5. Git коммит: `chore({slug}): откат задачи {N}, причина: {reason}`
6. Вывести: "Задача откачена. Запусти /ddrspec:run для продолжения"

### Документирование

### /ddrspec:doc-set {название}

Создать или обновить документацию для элемента проекта (компонент, сервис, модель).

1. Определить тип элемента (компонент, сервис, модель, контроллер, утилита)
2. Найти файл в кодовой базе
3. Проанализировать код
4. Создать документацию в `.claude/ddrspecs/docs/{type}/{name}.md`
5. Обновить `.claude/ddrspecs/docs/INDEX.json`
6. Вывести результат с предложениями задокументировать связанные компоненты

**Интеграция с задачами:**
- При упоминании `@ComponentName` в задачах читается документация из INDEX

**Структура документации:**
- Компоненты: props, emits, slots, примеры использования
- Сервисы: методы, параметры, возвращаемые значения, примеры
- Модели: структура, методы, связи
- Контроллеры: endpoints, методы, middleware
- Утилиты: функции, параметры, примеры

### /ddrspec:doc-list

Показать список всей документации проекта.

1. Прочитать `.claude/ddrspecs/docs/INDEX.json`
2. Для каждой сущности извлечь тип и краткое описание
3. Сгруппировать по типам (компоненты, сервисы, модели, контроллеры, утилиты, архитектура)
4. Вывести таблицу с количеством в каждой категории

**Пример вывода:**
```
# Документация DDRSpecs

## Компоненты (3)

| Название | Описание |
|----------|----------|
| BottomMenu | Компонент нижнего меню навигации. |
| MenuItem | Элемент меню с иконкой. |

---

**Всего:** 14 сущностей
```

### Инициализация

### /ddrspec:init

Инициализировать систему DDRSpecs в проекте (логирование + документирование).

1. Определить тип проекта (Laravel, React Native, Node.js, Python, Frontend SPA)
2. Настроить систему логирования:
   - Проверить существующую систему
   - Внедрить logger (если нет)

## Система логирования

**КРИТИЧЕСКИ ВАЖНО:** В проекте настроена система логирования. ВСЕГДА используй logger вместо console.log/console.error.

**Использование:**
```javascript
import logger from './src/utils/logger';

logger.error('Ошибка', { data: 'value' });
logger.warn('Предупреждение', { data: 'value' });
logger.info('Событие', { data: 'value' });
logger.debug('Отладка', { data: 'value' });
```

**Уровни:**
- `error` - критические ошибки
- `warn` - предупреждения
- `info` - важные события
- `debug` - отладка

**ВСЕГДА логируй:**
- Ошибки с полным stack trace
- API запросы/ответы
- Изменения состояния
- Критичные операции

**Логи:** `{documentDirectory}/app.log`

**Просмотр логов:** `/ddrspec:log`
