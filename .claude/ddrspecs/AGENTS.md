# DDRSpecs Framework

Микрофреймворк для управления задачами с экономией токенов и сохранением контекста между сессиями.

## Принципы

1. **Одна активная задача** — не распыляемся, контролируем контекст
2. **Минимум токенов** — читаем только нужные файлы (DISPATCHER → конкретная task)
3. **Git интеграция** — каждая задача = отдельная ветка
4. **Handoff** — можно продолжить в новом треде с того же места

## Структура

```
.claude/ddrspecs/
├── AGENTS.md                       # Этот файл (инструкции)
├── ACTIVE.md                       # Указатель на активную задачу
├── HISTORY.md                      # Глобальный контекст выполненных задач
└── {task-slug}/                    # Папка задачи
    ├── SPEC.md                     # Описание задачи
    ├── DISPATCHER.md               # Карта задач + статусы
    └── tasks/
        ├── 001-{name}.md           # Подзадача
        ├── 002-{name}.md
        └── ...
```

## ACTIVE.md

Содержит slug активной задачи:
```
drag-and-drop-filemanager
```

Если пустой или отсутствует — нет активной задачи.

## DISPATCHER.md — формат

```markdown
# {Task Name}

## Meta
- Slug: {task-slug}
- Branch: feature/{task-slug}
- Created: {date}
- Status: planning|in_progress|done

## Checkpoint
- Current: {номер текущей задачи или 0}
- Last Updated: {timestamp}

## Tasks
| # | File | Status | Title |
|---|------|--------|-------|
| 1 | 001-research.md | done | Исследование |
| 2 | 002-implement.md | current | Реализация |
| 3 | 003-test.md | pending | Тестирование |

## Context (кратко)
- Ключевое решение 1
- Ключевое решение 2
```

**Статусы задач:**
- `done` — выполнено
- `current` — в работе (только одна!)
- `pending` — ожидает
- `blocked` — заблокировано

## Task файл — формат

```markdown
# Task {N}: {Title}

## Цель
{Что нужно сделать — 1-2 предложения}

## Контекст
{Только критичная информация для этой задачи}

## План
1. [ ] Шаг 1
2. [ ] Шаг 2
3. [ ] Шаг 3

## Прогресс
{Заполняется по мере выполнения}

## Результат
{Заполняется после выполнения}

## Проблемы и решения
{Заполняется через /ddr:bug при возникновении проблем}

### 🐛 Проблема {N}: {название}
**Дата**: {YYYY-MM-DD HH:MM}
**Описание**: {описание проблемы}
**Решение**: {описание решения}
**Изменено**: {список файлов}
**Источники**: {ссылки на GitHub/StackOverflow}
```

## Команды

### /ddr:new {описание}

1. Уточняющие вопросы (AskUserQuestion)
2. Поиск best practices (Context7, WebSearch)
3. Предложение вариантов технологий
4. После выбора:
   - Создать папку `.claude/ddrspecs/{slug}/`
   - Создать SPEC.md, DISPATCHER.md, tasks/
   - Создать git ветку `feature/{slug}`
   - Записать slug в ACTIVE.md
5. Вывести: "Задача создана. Запусти /ddr:run для выполнения"

### /ddr:run

1. Прочитать ACTIVE.md → получить slug
2. Если пусто → "Нет активной задачи. Создай через /ddr:new"
3. Прочитать DISPATCHER.md → найти `current` задачу
4. Прочитать ТОЛЬКО файл текущей задачи
5. Выполнить задачу
6. Обновить статус в DISPATCHER.md:
   - Текущую → `done`
   - Следующую → `current`
   - Обновить Checkpoint
7. Если все задачи done → вывести "Все задачи выполнены. Запусти /ddr:done"

### /ddr:list

1. Прочитать все папки в `.claude/ddrspecs/`
2. Для каждой прочитать DISPATCHER.md (только Meta и Tasks)
3. Вывести таблицу:
```
# DDRSpecs Tasks

| Slug | Status | Progress | Branch |
|------|--------|----------|--------|
| drag-and-drop | in_progress | 2/4 ✓ | feature/drag-and-drop |
| auth-refactor | done | 5/5 ✓ | feature/auth-refactor |
```
4. Пометить активную задачу: `→ drag-and-drop (active)`

### /ddr:bug {описание проблемы}

Решить проблему в текущей задаче:

1. Прочитать ACTIVE.md → slug текущей задачи
2. Прочитать DISPATCHER.md → найти current task
3. Прочитать ТОЛЬКО файл текущей задачи
4. Анализировать проблему (логи, документация, @mentions)
5. Поиск решения (WebSearch, Context7)
6. Решить проблему
7. Обновить task файл — добавить секцию "Проблемы и решения":
   ```markdown
   ## Проблемы и решения

   ### 🐛 Проблема {N}: {название}
   **Описание**: {problem}
   **Решение**: {описание решения}
   **Изменено**: {список файлов}
   **Источники**: {ссылки}
   ```
8. Если решение критично → обновить Context в DISPATCHER.md
9. Git коммит: `fix({slug}): {описание}`
10. Обновить Checkpoint в DISPATCHER.md

**Когда использовать:**
- Ошибка при выполнении задачи
- Баг в коде
- Неожиданное поведение библиотеки
- Нужно изменить подход внутри задачи

### /ddr:done

Завершить активную задачу с git интеграцией и добавлением в глобальный контекст.

1. Прочитать ACTIVE.md → получить slug
2. Прочитать DISPATCHER.md — проверить статус всех задач
3. Если не все done → вывести список незавершенных, спросить подтверждение (AskUserQuestion)
4. Git операции (ПОСЛЕДОВАТЕЛЬНО через &&):
   - `git add . && git commit -m "feat({slug}): завершена задача {name}"`
   - `git checkout main`
   - `git merge feature/{slug} --no-ff`
   - Обработать ошибки (нет изменений, конфликты, нет ветки)
5. Обновить DISPATCHER.md: Status → done, Last Updated → timestamp
6. Добавить запись в `.claude/ddrspecs/HISTORY.md`:
   - Формат: дата, название, slug, branch, краткое описание (из SPEC.md)
   - ТОЛЬКО общее описание задачи (БЕЗ детализации по таскам)
   - Ключевые решения из Context (если есть)
   - Новые записи добавляются ВВЕРХ файла
7. Очистить ACTIVE.md
8. Вывести результат: "Задача завершена, смержена в main, добавлена в HISTORY.md"

**HISTORY.md формат:**
```markdown
### {YYYY-MM-DD}: {Task Name}
- **Slug**: {slug}
- **Branch**: feature/{slug}
- **Описание**: {краткое описание из SPEC.md}
- **Ключевые решения**: {Context из DISPATCHER.md}
---
```

**Важно:** Git команды выполнять строго последовательно, не параллельно!

### /ddr:redo

Отметить текущую задачу как нерешенную и вернуться к предыдущей задаче.

1. Прочитать ACTIVE.md → получить slug
2. Прочитать DISPATCHER.md → найти current task
3. Спросить причину через AskUserQuestion
4. Обновить DISPATCHER.md:
   - Текущую задачу → `pending`
   - Предыдущую задачу → `current`
   - Обновить Checkpoint (Checkpoint.Current -= 1)
5. Git коммит: `chore({slug}): откат задачи {N}, причина: {reason}`
6. Вывести: "Задача откачена. Запусти /ddr:run для продолжения"

## Алгоритм продолжения в новом треде

```
/ddr:run
    │
    ├── ACTIVE.md → "drag-and-drop-filemanager"
    │
    ├── DISPATCHER.md → Checkpoint.Current = 3
    │
    ├── tasks/003-implement.md (только этот файл!)
    │
    └── Продолжаем с шага, где остановились
```

**Экономия**: ~800 токенов вместо ~5000+

## HISTORY.md — глобальный контекст

**Назначение**: Хранить краткое описание ВСЕХ выполненных задач для экономии токенов.

**Когда читать:**
- В начале новой сессии (вместо чтения всех DISPATCHER.md)
- При необходимости вспомнить, что было сделано ранее
- Перед созданием новой задачи (чтобы не дублировать)

**Формат записей:**
```markdown
### {YYYY-MM-DD}: {Task Name}
- **Slug**: {slug}
- **Branch**: feature/{slug}
- **Описание**: {краткое описание цели из SPEC.md — 1-2 предложения}
- **Ключевые решения**: {Context из DISPATCHER.md, если есть}
---
```

**Правила:**
- Новые записи добавляются ВВЕРХ (после заголовка)
- ТОЛЬКО краткое описание цели задачи (БЕЗ детализации по таскам)
- Максимум 2-3 предложения на задачу
- Ключевые архитектурные решения (если критичны для понимания)

**Экономия токенов:**
- Вместо ~5000+ токенов (DISPATCHER.md + все tasks/*.md)
- Всего ~200-300 токенов на задачу в HISTORY.md

## Важные правила

1. **Не читать все task файлы** — только current
2. **Обновлять Checkpoint** после каждого действия
3. **Держать Context в DISPATCHER.md кратким** — только ключевые решения
4. **Git коммит** перед сменой задачи или завершением сессии
5. **HISTORY.md обновлять** только при завершении задачи через /ddr:done
